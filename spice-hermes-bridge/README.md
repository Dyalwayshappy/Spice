# Spice-Hermes Bridge

This bridge keeps Spice and Hermes separated.

Hermes receives external signals and coordinates execution. The bridge converts external signals and execution outcomes into structured Spice observations. Spice decides which observations matter, updates decision state, recommends actions, and absorbs decision-relevant execution outcomes.

## Architecture Boundary

Hermes handles ingress and execution.

The bridge only structures and validates observations and execution outcomes. It does not decide importance, update decision state, or perform conflict detection.

Spice decides what matters, updates `WorldState`, builds `ActiveDecisionContext`, produces recommendations, and absorbs decision-relevant execution outcomes.

## Bridge Boundary

The bridge does:

- Convert WhatsApp schedule messages into `commitment_declared` observations.
- Convert GitHub PR events into `work_item_opened` observations.
- Convert execution outcomes into `execution_result_observed` observations.
- Validate observation shape, provenance, and required identifiers.
- Deduplicate external delivery events where appropriate.

The bridge does not:

- Decide whether an observation is important.
- Decide whether an observation enters Spice `WorldState`.
- Maintain Spice decision state.
- Perform conflict detection.
- Store full WhatsApp chat history.
- Store Codex/Hermes transcripts, shell output, patches, or retry logs.
- Perform decision evolution.

For WhatsApp ingress specifically, the bridge is limited to:

- Input filtering.
- Information extraction.
- Extractor proposal validation.
- Uncertainty gating.
- Observation construction.
- Schema validation.

It does not judge whether a meeting is important, whether it affects a PR, whether it enters `WorldState`, or whether it should trigger a decision.

Spice owns:

- Observation ingestion.
- `WorldState`.
- `ActiveDecisionContext`.
- Conflict detection.
- Recommendation selection.
- Decision outcome absorption.

Hermes owns:

- Messaging ingress.
- External agent/tool execution.
- Execution transcript and tool detail.

## Observation Identity and Time

Adapters should create observations with the builder API, not by accepting external identifiers.

- `observation_id` is generated by the bridge as `obs_<uuid4 hex>`.
- `observation_id` is globally unique enough for bridge delivery, replay, and audit.
- `observed_at` is generated in UTC ISO-8601 format by default.
- Serialized observations must include `observed_at` with an explicit timezone offset.
- Source-local timestamps may be preserved in `attributes` or `provenance`, but `observed_at` remains the bridge observation timestamp.

For WhatsApp relative dates such as `今天`, `明天`, `后天`, and `大后天`:

- The anchor is the WhatsApp message metadata timestamp (`received_at`) when it is present and has a timezone.
- If `received_at` is absent or invalid, the bridge uses current system time in the configured `--timezone`.
- Naive `received_at` values are not used as anchors; the bridge records a warning and falls back to system time.
- Extracted times are serialized as ISO-8601 strings with an explicit timezone offset.
- Naive extracted datetimes are rejected by observation validation.

Example:

```json
{
  "observation_id": "obs_11111111111111111111111111111111",
  "observation_type": "commitment_declared",
  "source": "whatsapp",
  "observed_at": "2026-04-16T02:00:00+00:00",
  "confidence": 0.91,
  "attributes": {
    "summary": "投资人会议",
    "start_time": "2026-04-17T15:00:00+08:00",
    "duration_minutes": 120
  },
  "provenance": {
    "adapter": "whatsapp_schedule.v1"
  }
}
```

## Schema Shape

Observations use a strict base schema plus per-type attribute validation.

Base fields:

- `observation_id`
- `observation_type`
- `source`
- `observed_at`
- `confidence`
- `provenance`

Per-type `attributes` are validated for each supported `observation_type`:

- `commitment_declared`
- `work_item_opened`
- `executor_capability_observed`
- `execution_result_observed`

`attributes` is not a free-form dump. New fields should be added intentionally when Spice needs them for decision-relevant state.

For `commitment_declared`, the bridge emits a canonical shape when it builds an observation:

- `summary`
- `start_time`
- `end_time`
- `duration_minutes`
- optional `prep_start_time`
- optional `priority_hint`
- optional `flexibility_hint`
- optional `constraint_hints`

`start_time` must always be explicit. The bridge prefers both `end_time` and `duration_minutes` so future reducers can consume either representation without guessing. Schema validation still accepts one of `end_time` or `duration_minutes` for imported observations, but the WhatsApp adapter emits both when it can safely build an observation.

For `work_item_opened`, the bridge emits a canonical shape:

- `kind`: currently `pull_request`
- `repo`: GitHub `owner/name`
- `item_id`: PR number as a string
- `title`
- `url`
- `action`: currently `opened` or `review_requested`
- `event_key`: stable deduplication key
- optional `urgency_hint`
- optional `estimated_minutes_hint`
- optional `requires_attention`

The canonical dedup key lives in `attributes.event_key`. Provenance keeps GitHub API details such as `github_event_id`, `polled_at`, `api_source`, and `time_anchor_source`.

Future reducers should treat `kind`, `repo`, `item_id`, `title`, `url`, `action`, and `event_key` as the stable input contract. Provenance-only fields must not be required for reducer semantics.

For `executor_capability_observed`, the bridge emits a minimal runtime capability shape:

- `capability_id`: stable capability identifier.
- `action_type`: currently `delegate_to_executor`.
- `executor`: currently `codex` for the Hermes/Codex path.
- `supported_scopes`: supported execution scopes, such as `triage` or `review_summary`.
- `requires_confirmation`: whether human confirmation is required before execution.
- `reversible`: whether delegation can be reversed or abandoned safely.
- `default_time_budget_minutes`: default bounded execution budget.
- `availability`: currently `available` or `unavailable`.

This observation does not grant capability by itself. It lets Spice see capability as state. The demo domain then decides whether `delegate_to_executor` is eligible during candidate generation.

## Execution Outcome Contract

`execution_result_observed` carries the decision-relevant outcome of an execution attempt.

Required identifiers:

- `decision_id`: the unique recommendation identifier emitted by Spice.
- `execution_ref`: the unique execution attempt identifier emitted by Hermes or the execution adapter.

`execution_ref` is used for:

- Deduplicating repeated delivery of the same execution outcome.
- Distinguishing retries or multiple executors for the same `decision_id`.
- Replaying outcome history without storing full execution transcripts in Spice.

Hermes keeps how execution happened: tool calls, agent transcript, shell output, patch details, and retry flow. Spice receives only what changed in the world: status, acted item, elapsed time, blocking issue, risk change, follow-up need, and summary.

## WhatsApp Extractor Flow

WhatsApp ingress uses a stable three-state output:

```text
WhatsApp message
-> rule prefilter
-> extractor proposal
   - deterministic, or
   - llm_assisted
-> schema/field validation
-> uncertainty check
-> observation | pending_confirmation | ignored
```

The default extractor is `deterministic`.

`llm_assisted` is optional. The LLM is proposal-only:

- It may parse natural language into candidate fields.
- It may mark uncertain fields, assumptions, confidence, and whether confirmation is needed.
- It must not output final observations directly.
- It must not decide importance.
- It must not decide WorldState admission.
- It must not perform conflict detection.
- It must not trigger decisions or recommendations.

The bridge always runs proposal validation and gating before building an observation. LLM failure falls back to deterministic extraction or pending confirmation; it must not produce dirty observations.

LLM-assisted extraction is also checked against the original text. A proposal cannot become an observation when a precise `start_time` appears only in the proposal and is not supported by a precise time token in the WhatsApp message. For example, `明天下午有个会` must go to pending confirmation; the bridge must not silently turn `下午` into `15:00`.

To use command-backed LLM proposal extraction, set:

```bash
export SPICE_HERMES_LLM_COMMAND="path-to-your-proposal-command"
```

The command receives a JSON request on stdin and must return one JSON proposal on stdout. The bridge still owns observation IDs, timestamps, validation, and gating.

## Pending Confirmations

Unclear inputs do not enter Spice.

If a message is plausible but incomplete, the bridge returns `pending_confirmation` and stores it locally at:

```text
.spice-hermes/pending_confirmations.json
```

Example pending shape:

```json
{
  "result_type": "pending_confirmation",
  "observation": null,
  "pending_confirmation": {
    "pending_id": "pending_11111111111111111111111111111111",
    "message": "我理解你有一个日程，但缺少具体开始时间。请补充具体时间。",
    "missing_fields": ["start_time"],
    "uncertain_fields": ["start_time"],
    "assumptions": ["time_period_without_precise_start_time"],
    "original_text": "明天下午有个会",
    "source": "whatsapp",
    "created_at": "2026-04-16T02:00:00+00:00",
    "provenance": {
      "adapter": "whatsapp_perception.v2"
    }
  },
  "warnings": []
}
```

Pending confirmations are Bridge-local state. They are not observations and must not be sent to Spice until a later confirmed message yields a valid observation.

Stored pending confirmations include:

- `pending_id`: generated as `pending_<uuid4 hex>`
- `status`: `pending` or `resolved`
- `original_text`
- `followups`
- `missing_fields`
- `created_at`
- `resolved_at`
- `source`
- `provenance`

The bridge supports a minimal resolution flow:

```text
ambiguous WhatsApp input
-> pending_confirmation
-> user follow-up
-> bridge finds the most recent active pending in the same chat scope
-> original_text + followup_text
-> same extractor / validation / gating pipeline
-> observation or still pending
```

If the merged text yields a valid `commitment_declared` observation, the pending item is marked `resolved`, `resolved_at` is written, and the follow-up is recorded. If the merged text is still incomplete, the same pending item remains `pending` and the follow-up is appended.

Resolution is intentionally simple:

- Only the most recent unresolved pending item is considered.
- Matching is scoped by `chat_id` when available.
- The bridge does not do multi-pending disambiguation.
- The bridge does not do complex dialogue memory.
- The bridge does not infer user intent beyond `original_text + followup_text`.
- Resolved pending items are not reopened or modified.
- Repeating the same follow-up for an already resolved pending item is ignored in the resolve path.

Expiration/TTL and richer multi-turn disambiguation are not implemented yet.

## Gating Rules

The bridge only builds `commitment_declared` when the proposal is clear enough:

- event type is supported and explicit
- `start_time` is precise
- duration or end time is available, or a safe bounded default is allowed
- no uncertain critical fields
- no key assumptions from the extractor
- confidence is at or above the bridge threshold

The bridge returns `pending_confirmation` when:

- precise start time is missing
- duration/end time is missing for event types where default duration is unsafe
- critical fields depend on model assumptions
- the extractor marks uncertain fields
- confidence is low
- a proposed precise time is not supported by the original WhatsApp text

The bridge returns `ignored` for empty messages, commands, Hermes responses, unrelated chat, and near-miss meta questions.

Default duration is conservative:

- The bridge may default to 60 minutes only for bounded commitments such as meetings, calls, interviews, and appointments with an explicit start time.
- The bridge records `duration_source: default_safe` and emits a warning.
- Travel/departure/flight-like events do not get a silent default duration; they go to pending confirmation when duration/end time is missing.

Observation provenance records the extractor request and result:

- `extractor_mode`: requested mode, such as `deterministic` or `llm_assisted`
- `extractor`: actual proposal source used to build the observation
- `fallback`: whether LLM-assisted mode fell back to deterministic extraction
- `fallback_reason`: why fallback happened, if any
- `time_anchor_source`: `received_at` or `system_now`

## CLI Output Contract

`whatsapp-observe --json` always returns:

```json
{
  "result_type": "observation | pending_confirmation | ignored",
  "status": "observation_built | pending_confirmation | ignored",
  "reason": "string or null",
  "valid": true,
  "observation": {},
  "pending_confirmation": {},
  "warnings": [],
  "issues": []
}
```

Only one of `observation` or `pending_confirmation` is populated.

```bash

  --input-json examples/whatsapp_message.json \
```


Run an ambiguous message and write a pending confirmation:

```bash
cd spice-hermes-bridge
```

## GitHub PR Polling

GitHub polling is the second clean external signal line.

Flow:

```text
GitHub polling
-> fetch open PRs for one repo
-> normalize pull_request.opened / review_requested
-> build stable event_key
-> dedup against delivery_state
-> build work_item_opened observation
-> validate
-> append observations.jsonl audit
-> mark event_key processed
```

Supported v1 events:

- `pull_request.opened`
- `review_requested`

Both map to `work_item_opened`. `action` preserves the original normalized GitHub action.

`review_requested` is polling-based in v1. It means "the bridge first observed that this open PR currently has requested reviewers or teams." It does not claim to know the original GitHub review-request event timestamp.

Event key rule:

```text
github:<owner/name>:pull_request:<pr_number>:<action>
```

Examples:

```text
github:Dyalwayshappy/Spice:pull_request:123:opened
github:Dyalwayshappy/Spice:pull_request:123:review_requested
```

The same `event_key` is processed only once. The bridge marks an event processed only after a valid observation is built and appended to the local audit log. If build or validation fails, the event is not marked processed.

If the audit log was written but delivery state was not updated, the next polling pass repairs `delivery_state.json` from `observations.jsonl` and skips emitting a duplicate observation.

Delivery state:

```text
.spice-hermes/delivery_state.json
```

Shape:

```json
{
  "processed_event_keys": {
    "github:Dyalwayshappy/Spice:pull_request:123:opened": {
      "processed_at": "2026-04-17T00:20:00+00:00",
      "observation_id": "obs_11111111111111111111111111111111"
    }
  }
}
```

Observation audit:

```text
.spice-hermes/observations.jsonl
```

Each line is one normalized observation. This is an audit/delivery artifact for later ingest; it is not Spice `WorldState`.

Observed time rule:

- Prefer GitHub event time when available, currently PR `created_at` for `opened`.
- Use poll time when GitHub does not provide an event timestamp, currently `review_requested` from the pulls API.
- `observed_at` is always ISO-8601 with timezone, normalized to UTC.
- `provenance.time_anchor_source` is `github_event_time` or `poll_time`.

For `review_requested`, `observed_at` is the polling time and `provenance.time_anchor_source` is `poll_time`.

GitHub hint fields:

- `urgency_hint` currently uses a fixed conservative default.
- `estimated_minutes_hint` currently uses a fixed conservative default.
- `requires_attention` is true because the normalized event represents an outstanding work item, not because the bridge decided it is important.

These fields are lightweight hints for downstream Spice reducers. They are not importance judgment, conflict detection, or recommendation logic.

The GitHub adapter does not decide whether a PR is important, whether it conflicts with a commitment, whether it enters `WorldState`, or whether it should trigger a recommendation. Spice owns those decisions.

Run a GitHub polling pass:

```bash
cd spice-hermes-bridge
python -m spice_hermes_bridge.cli poll-github \
  --repo Dyalwayshappy/Spice \
  --json
```

If needed, set a GitHub token:

```bash
export GITHUB_TOKEN="github_pat_or_token"
python -m spice_hermes_bridge.cli poll-github \
  --repo Dyalwayshappy/Spice \
  --json
```

`poll-github --json` returns:

```json
{
  "result_type": "poll_result",
  "status": "ok | partial | error",
  "repo": "Dyalwayshappy/Spice",
  "observations_built": [],
  "deduped_event_keys": [],
  "warnings": [],
  "issues": []
}
```

## Local Spice Demo Glue Flow

The bridge includes a thin integration path that connects the bridge, the Spice `examples/decision_hub_demo`, and Hermes/Codex through SDEP.

Flow:

```text
bridge observations
-> Spice demo reducer
-> WorldState / ActiveDecisionContext
-> simulation-driven recommendation
-> confirmation request when required
-> SDEP execute.request
-> Hermes SDEP wrapper
-> SDEP execute.response
-> execution_result_observed
-> Spice demo reducer
-> WhatsApp-ready reply text
```

The glue modules live under:

- `spice_hermes_bridge.integrations.spice_demo`: bridge observation to Spice demo ingestion and SDEP-backed control flow.
- `spice_hermes_bridge.integrations.hermes_sdep_agent`: Hermes/Codex SDEP executor endpoint.
- `spice_hermes_bridge.integrations.hermes_sdep_native`: native Hermes/Codex execution normalization behind the SDEP wrapper.
- `spice_hermes_bridge.integrations.whatsapp_reply`: WhatsApp-facing text formatting for confirmation and execution results.
- `spice_hermes_bridge.integrations.hermes_executor`: legacy/debug direct executor override, not the public default path.

Run the local flow through the canonical SDEP-backed execution spine:

```bash
PYTHONPATH=spice-hermes-bridge:. python -m spice_hermes_bridge.cli run-demo-flow \
  --choice confirm \
  --json
```

Explicit debug overrides are still available for local testing, but they are not
the public default path:

```bash
PYTHONPATH=spice-hermes-bridge:. python -m spice_hermes_bridge.cli run-demo-flow \
  --executor mock \
  --choice details
```

Confirmation choices:

- `confirm`: create an execution request, send SDEP `execute.request`, normalize the SDEP response, emit `execution_result_observed`, and update Spice state through the reducer.
- `reject`: do not execute and do not create an execution outcome.
- `details`: return the decision trace summary and do not execute.

Current boundary:

- The command simulates the WhatsApp reply choice locally.
- It does not install a WhatsApp auto-reply service.
- It does not add GitHub webhooks.
- It does not persist a production confirmation store.
- Execution transcripts stay with Hermes/Codex; Spice only receives the structured outcome.

## WhatsApp Gateway Glue

The bridge also exposes a minimal WhatsApp webhook app for the local Spice + Hermes control loop.

Flow:

```text
WhatsApp message
-> webhook handler
-> WhatsApp perception ingress
-> Spice decision_hub_demo reducer
-> recommendation / confirmation
-> SDEP execute.request
-> Hermes SDEP wrapper
-> SDEP execute.response
-> execution_result_observed
-> Spice state update
-> WhatsApp-facing reply
```

Run the webhook server:

```bash
PYTHONPATH=spice-hermes-bridge:. \
  uvicorn spice_hermes_bridge.integrations.whatsapp_server:app \
  --host 0.0.0.0 \
  --port 8000
```

Webhook endpoint:

```text
POST /whatsapp/webhook
```

Accepted minimal payload:

```json
{
  "chat_id": "whatsapp.self",
  "sender": "me",
  "text": "今天下午3点有个会"
}
```

The server also accepts common nested gateway payloads where the message is under `data`, `payload`, `event`, or `message`.

Message routing:

- `1`, `同意`, or `confirm`: resolve the latest pending confirmation for that chat and execute.
- `2`, `拒绝`, or `reject`: reject the latest pending confirmation for that chat and do not execute.
- `3`, `详情`, or `details`: return the decision explanation for that chat and do not execute.
- Other text: run the WhatsApp perception pipeline and, when an observation is produced, ingest it into the Spice demo domain.

For each chat, the v1 server keeps one process-local Spice demo session and one active confirmation. This is intentionally small and is not a production multi-user session system.

Reply sending:

- By default, replies use a dry-run sender and are returned in the HTTP response.
- To forward replies through a local Hermes WhatsApp send endpoint, set:

```bash
export HERMES_WHATSAPP_SEND_URL="http://127.0.0.1:3000/send"
export HERMES_WHATSAPP_SEND_TOKEN="optional-token"
```

Hermes' local WhatsApp bridge send endpoint accepts:

```json
{
  "chatId": "whatsapp.self",
  "message": "reply text"
}
```

Configure the Hermes WhatsApp gateway, webhook forwarder, or local tunnel to deliver inbound messages to:

```text
http://127.0.0.1:8000/whatsapp/webhook
```

If the gateway requires an externally reachable URL, expose the server with a tunnel such as ngrok and use the HTTPS URL that forwards to port `8000`.

### WhatsApp Inbound Relay + Public Tunnel

Hermes' local WhatsApp bridge exposes:

- `GET /messages`: drains inbound WhatsApp messages from the local bridge queue.
- `POST /send`: sends a WhatsApp reply using `{"chatId": "...", "message": "..."}`.

The bridge relay polls `GET /messages` and forwards each inbound message to the Spice webhook:

```text
WhatsApp
-> Hermes gateway / local WhatsApp bridge
-> spice_hermes_bridge.integrations.hermes_relay
-> WHATSAPP_WEBHOOK_URL
-> local webhook server
-> Spice decision_hub_demo
-> SDEP execute.request
-> Hermes SDEP wrapper
-> SDEP execute.response
-> HERMES_WHATSAPP_SEND_URL
-> WhatsApp reply
```

Run everything with the helper script:

```bash
scripts/start_all.sh
```

This starts:

- the local webhook server on `0.0.0.0:8000`
- an ngrok tunnel to port `8000`
- the Hermes inbound relay

It prints:

```text
Public URL: https://xxxx.ngrok-free.app
Webhook endpoint: https://xxxx.ngrok-free.app/whatsapp/webhook
```

Manual three-terminal setup:

```bash
# Terminal 1: local webhook server
PYTHONPATH=spice-hermes-bridge:. \
  uvicorn spice_hermes_bridge.integrations.whatsapp_server:app \
```


Expected user-facing flow:

- Send a schedule message in WhatsApp, for example `明天下午3点有个会`.
- The webhook records a clean observation or asks for missing details.
- If a work item conflict exists, Spice returns a confirmation message with `1 同意执行`, `2 拒绝`, `3 查看详情`.
- Reply `1` to confirm execution, or `2` / `3` to reject or inspect details.

Current boundary:

- The server does not decide importance, conflict, or recommendation itself.
- The server auto-ingests a minimal Hermes/Codex `executor_capability_observed` observation so the demo domain can see delegation capability as state.
- The server does not persist Spice `WorldState` across process restarts.
- The server does not add webhook authentication yet.
- The server does not implement GitHub webhooks; GitHub PRs still enter through the existing polling adapter or test/demo seeding.
- The reply sender depends on a Hermes-compatible send endpoint. Without one, the HTTP response still contains the exact text that should be sent to WhatsApp.
- The inbound relay is a local polling bridge, not a production messaging queue.
